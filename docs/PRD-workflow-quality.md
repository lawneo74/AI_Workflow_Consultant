# PRD — Workflow Quality Release

**Product:** AI Workflow Architect
**Author:** Product Management
**Status:** Approved for build
**Release theme:** Ground the planner in real tool knowledge, guarantee the *simplest* workflow that gets the job done, and let users bring their own context.

---

## 1. Problem statement

The app already produces chained, copy-paste workflows, but three gaps limit plan quality:

1. **Shallow tool knowledge.** Routing decisions rest on one-line tool descriptions hardcoded in `app.py`. The team maintains a much richer capabilities write-up (strengths, output formats, anti-fits, proven tool sequences) that the planner never sees — and that write-up changes as the tools evolve.
2. **No simplicity guarantee.** The generator produces one plan per run. For non-trivial tasks there is no exploration of alternatives, so users sometimes receive a heavier chain than the task needs. Simplicity is the product's core promise: the *least* workflow that fully delivers the goal.
3. **Context is text-only.** Users often have a brief, an outline, or an existing deck that defines the task, but can only paste text into a box. Without that context the plan is generic.

Two supporting quality gaps: complex plans are generated by Sonnet when Opus-class reasoning is available, and the generated step prompts — the product's actual deliverable — must consistently meet the bar of a senior professional prompt engineer.

## 2. Goals / non-goals

**Goals**

- G1. Every routing and sequencing decision is grounded in the maintained tool-capabilities write-up, which non-developers can update in the GitHub UI without touching code.
- G2. Simple tasks yield the minimal viable workflow; non-simple tasks are planned as ≥2 distinct candidates, with the simplest goal-complete candidate selected and presented.
- G3. Users can attach `.md`, `.docx`, and `.pptx` files as planning context.
- G4. Complex-task generation runs on Claude Opus 4.8.
- G5. Every generated step prompt reads as authored by a senior prompt engineer.

**Non-goals**

- Expanding the 5-tool picker (Claude Code and Perplexity modes remain step-level *mode* recommendations, not separately selectable tools).
- Executing generated prompts, or any change to the hard PRD constraints (no rate-limit warnings, no MCP setup guides, no vertical prioritization, minimal UI).
- Showing multiple workflows to the user — candidates are internal; only the winner is presented.

## 3. Requirements

### F1 — Tool-capabilities knowledge base (G1)

- `tool_capabilities.md` lives in the repo root; content is the maintained capabilities write-up (comparison table + combination sequences).
- Loaded at app startup and injected into the generator and reviewer system prompts. If the file is missing or unreadable, the app falls back to the built-in catalog descriptions and continues (best-effort, consistent with the app's error philosophy).
- The file opens with a maintainer note: edit directly on GitHub (pencil icon → commit); the app picks the change up on next restart with no code change.
- Capability detail beyond the 5 picker tools (e.g. Claude Code, Perplexity Deep Research) surfaces inside steps via the `model` (mode) field and prompt text — never as a routed `app` value.

### F2 — Simplest-workflow selection (G2)

- **Simple tasks:** one workflow, explicitly minimal — a single step/tool whenever that fully delivers the goal.
- **Non-simple tasks:** the generator returns **2–3 genuinely different candidate workflows in one structured call** (different tool sequences or strategies, not cosmetic variants). A selection pass chooses the simplest candidate that fully achieves the stated goal — goal completeness is a hard gate; among complete candidates, fewest steps, then fewest tool switches, wins.
- Only the selected workflow proceeds to review and display. The review summary states that alternatives were considered and why the winner is the simplest sufficient plan.
- If candidate generation or selection fails, fall back to today's single-workflow path rather than failing the run.

### F3 — Context attachments (G3)

- Multi-file uploader accepting `.md`, `.docx`, `.pptx` beside the task box.
- Text extracted with pure-Python libraries (`python-docx`, `python-pptx`; `.md` read as text). Per-file and total character caps with visible truncation notes; extraction failure warns and continues.
- Extracted context is provided to the router, generator/selector, and reviewer. Attachment filenames are recorded in the plan header and all exports.
- Changing attachments invalidates a previously generated workflow (same rule as editing the task text).

### F4 — Opus 4.8 for complex tasks (G4)

- `COMPLEX_GENERATOR_MODEL = "claude-opus-4-8"` with adaptive thinking enabled explicitly (Opus 4.8 does not think when the field is omitted) and the correct 128K output-cap entry. Simple tasks stay on Haiku; the reviewer stays on Sonnet 5 as an independent, cost-effective quality gate.

### F5 — Senior-prompt-engineer output bar (G5)

- The prompt-engineering principles baked into the generator/selector/reviewer are rewritten to a senior professional standard: precise role framing, explicit deliverable and format specification, context carried across steps, constraints and edge-case guidance, self-check/quality criteria, and audience calibration.
- The reviewer enforces this bar as an explicit checklist item and rewrites any step prompt that falls short.

## 4. Acceptance criteria

1. With `tool_capabilities.md` present, generator/reviewer system prompts contain its content; with the file removed, the app still runs on fallback descriptions.
2. A trivially simple task produces a 1-step (or clearly minimal) plan; a non-simple task's run produces ≥2 internal candidates and presents exactly one workflow.
3. `.md`, `.docx`, `.pptx` files upload, are extracted, appear in the context sent to the planner, and are listed in Markdown/DOCX/PDF exports; oversized files are truncated with a note, and corrupt files warn without aborting.
4. Complex runs call `claude-opus-4-8`; simple runs call `claude-haiku-4-5`; no API call exceeds its model's output-token cap.
5. Existing behaviors regress nowhere: auth, research step, review fallback, result invalidation, session reset, and all three export formats pass the AppTest suite.

## 5. Risks & mitigations

- **Latency/cost of multi-candidate generation** — one structured call (not N calls) for candidates; selection reuses the reviewer-class model with a small output. Simple tasks are unaffected.
- **Attachment content blows the input budget** — hard caps with truncation notes; caps chosen well inside the 1M-context models' limits.
- **Capabilities file edited into an unusable state** — the file is prompt context, not parsed config; malformed Markdown degrades quality but cannot crash the app. Missing file falls back to built-ins.
